# Output Formats

Use the lightest format that still helps the user decide.

## 1. Quick shortlist

Best when the user wants fast recommendations.

Suggested structure:
- brief assumptions or scope note
- 3 to 7 recommended repos
- one short paragraph each covering what it is and why it fits
- short final recommendation

## 2. Comparison table

Best when the user wants side-by-side tradeoffs.

Suggested columns:
- Repo
- Type
- Why it fits
- Risks or gaps
- Language and license
- Link

## 3. Categorized landscape

Best when the search space mixes different repo roles.

Suggested structure:
- Products
- Frameworks
- Infrastructure
- Curated lists
- Research or experiments

Within each category, include a short note on when to choose it.

## 4. Research memo

Best when the user wants a more methodical scan.

Suggested structure:
1. Request summary
2. Search strategy used
3. Filters and assumptions
4. Shortlist
5. Notes on tradeoffs
6. Recommended next inspection steps

## Final recommendation patterns

When useful, end with one of these:
- If you want something deployable first, start with X.
- If you want something easiest to modify, start with Y.
- If you want the clearest architecture reference, read Z first.
- If you want broad discovery rather than implementation, begin with the curated list.
